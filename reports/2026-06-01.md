# Daily Report - 2026-06-01

## Summary

Implemented cron-based activity scheduling in the `activities` app. This was previously
a placeholder (`RecurrenceType.CRON` existed but was labeled "Cron (future)" with no
implementation). The feature is now fully functional.

## Changes Made

### Model
- Added `cron_expression` field (`CharField(100)`, nullable) to `Activity`
- Implemented `_compute_cron_next()` — a built-in 5-field cron parser that computes
  the next matching datetime from an expression
- Updated `save()` to set `next_due_at` from `cron_expression` when creating cron activities
- Renamed `RecurrenceType.CRON` label from `"Cron (future)"` to `"Cron"`
- Migration: `activities/migrations/0003_add_cron_expression.py`

### Serializer
- Added `cron_expression` to `ActivitySerializer` and `ActivityCreateSerializer` fields
- Validation: `cron_expression` is required when `recurrence_type='cron'`
- Validation: Invalid cron expressions are rejected with a clear error message

### Services
- Added `reschedule_recurring()` — after successful execution, computes the next
  cron match and transitions the activity back to `PENDING` for scheduler re-pickup

### Celery Beat
- Wired `activities.scheduler.poll` into `CELERY_BEAT_SCHEDULE` (every 60s)
- Wired `activities.recover_stale` into `CELERY_BEAT_SCHEDULE` (every 300s)

### Tests
- 9 new tests covering: serializer validation, model save, API create, invalid
  expression rejection, reschedule_recurring, and non-recurring passthrough

### Documentation
- Updated `activities/README.md` — cron is no longer "future", API docs reflect
  `cron_expression`, new example for cron activity creation

## Verification

- Ruff: all checks pass
- MyPy: no issues
- Pytest: all 9 new tests pass
