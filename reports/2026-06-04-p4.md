# Daily Report — 2026-06-04 (Phase 4 Ship)

## Summary

Shipped **Phase 4 (Podcasts)** of the radio roadmap. The new top-level
`podcasts/` Django app delivers an RSS-driven podcast catalogue and
per-episode audio URLs to the existing Django + Nextcloud ecosystem.
Commit `b77c0c7e feat(podcasts): add RSS-driven podcast catalogue and
ingestion (Phase 4)` is on `main` (27 files, +2308/−5).

The earlier Phase 3 commit (`21b74385`, favorites + listening history)

## What shipped

- **New app `podcasts/`** — top-level Django app, registered in
  `INSTALLED_APPS`, mounted at `/api/v1/podcasts/`.
- **Two models** (`podcasts/models.py`):
  - `Podcast` (slug PK, unique `feed_url`, refresh status fields).
  - `PodcastEpisode` (FK to `Podcast`, `unique(podcast, guid)`,
    composite `(podcast, -published_at)` index).
- **Service layer** (`podcasts/services.py`): `ingest_podcast`,
  `fetch_feed`, `parse_feed_bytes`, `_coerce_published`
  (`email.utils.parsedate_to_datetime` for RFC-822; Django's
  `parse_datetime` / `parse_date` for ISO-8601 and date-only),
  `_coerce_duration` (seconds, `MM:SS`, `HH:MM:SS`),
  `ingest_all_active_podcasts`, `list_episodes_for_podcast`,
  `IngestionReport` dataclass, `PodcastIngestionError`.
- **Celery task** `podcasts.tasks.refresh_all_feeds` with
  `time_limit=900`, `soft_time_limit=850`, scheduled hourly via beat
  (`PODCASTS_REFRESH_INTERVAL_SECONDS=3600`).
- **5 endpoints** (`podcasts/views.py` + `urls.py`):
  - `GET /api/v1/podcasts/` — list active podcasts.
  - `GET /api/v1/podcasts/<slug>/` — podcast details.
  - `GET /api/v1/podcasts/<slug>/episodes/?limit=N` — episode list
    (1..500, default 100, newest first).
  - `GET /api/v1/podcasts/episodes/<id>/stream/` — audio URL +
    metadata.
  - `POST /api/v1/podcasts/<slug>/refresh/` — auth'd, throttled
    `5/min`.
- **OpenAPI** regenerated: 5 new operation IDs (`v1_podcasts_*`) with
  envelope schemas matching `config.api.responses.success_response`.
- **Sample data** via `python manage.py load_sample_podcasts` (BBC
  Global News + NPR Up First; idempotent).
- **Tests** (31 new): `test_models.py` (4), `test_services.py` (14),
  `test_views.py` (13).
- **Docs** aligned: new `podcasts/README.md`; `IMPLEMENTATION_SUMMARY.md`
  Phase 4 section with file:line evidence; `05_data_model.md` extended
  with `Podcast` and `PodcastEpisode` and a new Mermaid entity;

## Architectural decisions

1. **Top-level app, not nested under `radio/`.** Podcasts are a
   separate content domain with their own ingestion cadence, schema,
   and operational profile; nesting them under `radio/` would have
   muddled boundaries without saving code.
2. **`httpx` + `feedparser`.** `httpx` matches the existing project
   convention; `feedparser` is the de-facto Python RSS/Atom parser
   and is maintained widely.
3. **Best-effort ingestion, never raises.** A single bad feed is
   recorded on `Podcast.last_refresh_status` and
   `last_refresh_error` and never blocks the periodic task. This
   matches the radio Phase 2 health-check failure isolation pattern.
4. **`unique(podcast, guid)`** allows the same `guid` to appear
   under different podcasts without collision; a few podcast
   directories re-use short stable strings.
5. **RFC-822 date support via `email.utils.parsedate_to_datetime`.**
   Django's `parse_datetime` does not handle RFC-822; most RSS feeds
   emit RFC-822. We fall back to ISO-8601 (`parse_datetime`) and then
   date-only (`parse_date`) for robustness.
6. **1-hour refresh cadence.** Matches typical RSS churn for news
   podcasts; configurable via `PODCASTS_REFRESH_INTERVAL_SECONDS`.
7. **No audio proxying, per ADR 002.** We expose the upstream audio
   URL through the standard envelope; the client streams directly.
