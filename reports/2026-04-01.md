Report: Redis Sentinel monitoring + Celery failover drill (as of Apr 1, 2026)

Scope of work

- Re-pointed `redis_exporter` to the local Redis Sentinel endpoint.
- Verified Prometheus is scraping Sentinel metrics rather than standalone Redis metrics.
- Ran a second failover drill against the currently elected master and measured Celery task behavior during the outage window.

Findings

- Monitoring is now Sentinel-aware:
  - `.env` sets `REDIS_ADDR=redis://host.docker.internal:26379`.
  - `redis_exporter` reports `redis_mode="sentinel"` and `tcp_port="26379"` via `redis_instance_info`.
  - Prometheus exposes Sentinel metrics from the exporter, including:
    - `redis_sentinel_master_status`
    - `redis_sentinel_master_ok_sentinels`
    - `redis_sentinel_master_ok_slaves`
    - `redis_sentinel_masters`
- Metrics followed failover correctly:
  - Before the second drill, `redis_sentinel_master_status` reported `master_address="172.20.0.3:6379"`.
  - After stopping `redis-replica`, Sentinel elected `172.20.0.2:6379` on the first poll.
  - Prometheus reflected the new `master_address="172.20.0.2:6379"` after the next scrape.
- Celery survived failover, but not quickly:
  - A task submitted during failover (`9b57ba3b-a707-4c13-9255-d74de411b64b`) stayed `PENDING` for about `54.7s`.
  - The same task eventually returned `SUCCESS`.
  - A fresh task after recovery completed normally.

Assessment

- Sentinel failover and Prometheus visibility are verified.
- Django cache behavior during failover is verified from the earlier Phase 3 drill.
- Celery behavior is acceptable only for background workloads where roughly one minute of queueing delay during broker failover is tolerable.
- Celery behavior is not acceptable for latency-sensitive task dispatch without further reconnect tuning and another drill.

Recommended follow-up

- If the operational target is "task dispatch continues within ~10-15s of Sentinel promotion", tune Celery reconnect behavior and rerun the failover drill with timing capture.
- If the operational target is limited to eventual completion for background jobs, the current setup is sufficient and Phase 3 can be treated as complete with the documented latency caveat.
