  Part 1: Single Source of Truth — `classify_status_code()`

  Problem: HTTP status-code-to-retry-category mapping was duplicated in
  4 places with subtle inconsistencies:

    1. `ndvi/retry_policy.py` — `_status_retry_classification()`
    2. `ndvi/stac_client.py` — `_retry_category_for_status()` (defined TWICE)
    3. `ndvi/engines/sentinelhub.py` — inline if/elif in `SentinelHubUpstreamError`
    4. `ndvi/raster/sentinelhub_engine.py` — inline if/elif in `SentinelHubRasterError`

  Fix:
    - Added `classify_status_code()` in `ndvi/retry_policy.py` as the
      canonical, documented truth table for all status code classification.
    - Removed all duplicated implementations.
    - All error types now delegate to this single function.

  Truth table (canonical):

    ============================= ========= ==========================
    Status Code                   Retryable Category
    ============================= ========= ==========================
    401, 403                      False     AUTH
    400, 422                      False     VALIDATION
    429                           True      RATE_LIMIT
    >= 500                        True      TRANSIENT_UPSTREAM
    Anything else / None          False     UNKNOWN
    ============================= ========= ==========================

  Part 2: Unified Exception Hierarchy

  Made all NDVI upstream error types inherit from `UpstreamFailureError`
  (which inherits from `NdviFailureError`), ensuring they carry consistent
  `retryable`, `category`, `status_code`, and `delay` attributes:

  - `StacUpstreamError`: Now inherits `UpstreamFailureError + StacError`.
    Constructor delegates to `classify_status_code()`.
  - `SentinelHubUpstreamError`: Now inherits `UpstreamFailureError`.
    Constructor delegates to `classify_status_code()`.
  - `SentinelHubRasterError`: Now inherits `UpstreamFailureError`.
    Constructor delegates to `classify_status_code()`.
  - `SentinelHubAuthError`: Now inherits `UpstreamFailureError` with
    `retryable=False, category=AUTH` (previously plain `RuntimeError`).

  This enables `should_retry()` to make consistent retry decisions across
  all engines and error types.

  Part 3: Network Error Handling Fix (Medium Severity)

  Problem: After inline retry exhaustion on `httpx.RequestError` (network
  timeout, DNS failure, connection refused), both SentinelHub engines did
  a bare `raise`. This propagated a raw `httpx.RequestError` to the Celery
  task handler, where `should_retry()` returned `retry=False` (not an
  `NdviFailureError` subclass). Result: transient network errors were
  silently dropped instead of being retried.

  Fix:
    - `ndvi/engines/sentinelhub.py`: After retry exhaustion on
      `httpx.RequestError`, now raises `SentinelHubUpstreamError(None, ...)`
      which `classify_status_code` maps to `retryable=False, UNKNOWN`.
    - `ndvi/raster/sentinelhub_engine.py`: Same fix — raises
      `SentinelHubRasterError(None, str(exc))` after retry exhaustion.

  Note: `status_code=None` maps to `retryable=False` because the network
  error has already been retried inline (3 attempts). The Celery-level
  retry decision is deferred to the task handler based on job context.
