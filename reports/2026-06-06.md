## Detailed change log

### #4 — `MEDIA_ROOT` + storage backend (`ea89e020`)

**Before.** `AudioAlert.audio_file` `FileField` had nowhere to
write; `MEDIA_ROOT` and `MEDIA_URL` were unset; no S3-friendly
swap knob.

**After.** `MEDIA_URL`, `MEDIA_ROOT` (default
`<BASE_DIR>/var/media`; test `<BASE_DIR>/var/media-test`),
`DEFAULT_FILE_STORAGE` env-overridable. `.gitignore` excludes
`var/` and `audio_alerts/`.

### #5 — Metrics + SLO Prometheus rules (`d16cb886`)

**Before.** App emitted zero metrics. Operators could not
answer "how many alerts did we push in the last 5 minutes"
without parsing logs.

**After.**

- `alerts/metrics.py` — 8 series: dispatch total / failures /
  duration, render duration / failures, TTS circuit state,
  push attempts / failures.
- `podcasts/metrics.py` — 3 series: refresh total / duration,
  refresh_stale.
- New Prometheus rule groups: `alerts` and `podcasts` (see
  `monitoring/prometheus/alerts.yml`).

### #10 — Request-id / correlation-id (`4a255de0`)

**Before.** Every response was anonymous. An operator tracing
a 500 had to grep on timestamps.

**After.**

- `config/api/request_id.py` — `RequestIdMiddleware` +
  contextvar helpers (`current_request_id`, `bind_request_id`,
  `reset_request_id`, `new_request_id`).
- Middleware reads `X-Request-ID` from the incoming request or
  generates a UUID4; the same value is exposed on the
  response as `X-Request-ID` and embedded in the envelope
  (`success_response` / `error_response` /
  `custom_exception_handler`).
- `config/celery.py` propagates the active id into Celery task
  headers and the AMQP `correlation_id`; the task body binds
  it back into the `ContextVar` on `task_prerun` and restores
  on `task_postrun`.
- OpenAPI schema updated; Swagger UI shows `request_id` on
  every envelope.

### #8 — WebSocket push metrics + alert (`bb5a9d24`)

**Before.** `emit_audio_alert_event` returned an int but it
was thrown away in production. No way to alert on a silent
push outage.

**After.** `alerts/services.py:emit_audio_alert_event` records
`alerts_push_attempts{result=no_layer|success|error}` and
`alerts_push_failures{reason=ExceptionClass}`. The
`WebSocketPushFailureRateHigh` rule fires when failures
exceed 10% over 5 min.

### #7 — TTS circuit breaker + concurrency cap (`ee66f41c`)

**Before.** A misbehaving TTS engine could block workers
indefinitely. No per-engine rate cap, no load shedding.

**After.** `alerts/tts_breaker.py`:

- `TTSCircuitBreaker` opens after 5 failures / 60s, stays
  open 60s, then half-open probes a single call.
  `TTSCircuitOpenError` (renamed from `TTSCircuitOpen` for
  N818).
- `TTSExecutorPool` — per-engine `ThreadPoolExecutor` with
  default 4 workers, env-overridable via `TTS_MAX_WORKERS`.
- `_breakers_lock` is `threading.RLock` (plain `Lock` would
  deadlock on `get_pool → get_breaker`).
- `alerts/apps.py:ready()` probes engines at startup.

### #6 — Split dispatch into fast + render tasks (`b9cceb8c`)

**Before.** `dispatch_alert` did everything synchronously:
row insert, push, TTS synthesis, audio save. HTTP path could
spend seconds in TTS.

**After.**

- `dispatch_alert_fast` writes the row + pushes the initial
  WS event, returns immediately. Canonical entry for
  triggers.
- `dispatch_alert` is a thin wrapper that calls the fast path
  and enqueues `render_alert_audio`.
- `render_alert_audio` is the async render (autoretry 3 with
  backoff + jitter, idempotent).

### #3 — Parallel podcast ingestion (`9d77aa46`)

**Before.** `refresh_all_feeds` processed feeds serially; a
failing feed had no backoff.

**After.**

- New beat entry `dispatch_refresh_batch` lists due feeds
  and builds a Celery `chord` with one
  `refresh_one_podcast.s(pid)` per show and a
  `summarise_refresh_run` callback.
- `Podcast.next_retry_at` + `consecutive_failures` drive
  per-feed backoff; partial index on
  `(is_active, next_retry_at)` keeps the due-set query
  cheap.
- Backoff schedule: 1m, 5m, 1h, 24h
  (`_BACKOFF_SECONDS = (60, 300, 3600, 86_400)`).
- New `podcasts_ingestion` queue with task routes; legacy
  `refresh_all_feeds` kept for one-off management commands.

### #1 — Admin broadcast parallelism (`c3647a8f`)

**Before.** `on_admin_broadcast` ran `dispatch_alert`
synchronously in a for-loop; one slow TTS render blocked
every other recipient.

**After.** Fans out via a `celery.group` of
`dispatch_one_alert` tasks (one per recipient, default
queue). The new task:

- `autoretry_for=(OperationalError, DatabaseError,
  ConnectionError, TimeoutError)`, with
  `dont_autoretry_for=(ValueError,)` so transient infra
  failures self-heal but a programming error surfaces
  immediately.
- `max_retries=2`, `retry_backoff=True`, `retry_jitter=True`.
- `acks_late=True`.

The trigger falls back to a synchronous in-process fan-out if
the Celery broker is unreachable, so a Redis outage never
silently drops a broadcast. View contract
(`{"data": {"dispatched": <count>}}`) unchanged.

### #2 — Retention policy + purge tasks (`ece7e89f`)

**Before.** `AudioAlert`, `ListeningHistory`,
`StationHealthCheck` grew without bound. Audio files
accumulated on disk.

**After.** Four new Celery Beat tasks, all daily at 05:xx UTC
and spread out by 15 minutes to avoid thundering-herd
contention:

- `alerts.tasks.purge_old_alerts` (default 90 days)
- `alerts.tasks.purge_orphan_audio_files` (safety-net
  walker)
- `radio.tasks.purge_old_history` (default 90 days)
- `radio.tasks.purge_old_health_checks` (keep N newest per
  station; default 20)

`docs/architecture/radio/09_operational.md` has a new
**Retention Policy** section documenting settings, tasks,
schedules, and operator verification steps.

### #9 — DB-aware `alert_types` filter (`f836775a`)

**Before.** The JSON-array filter was always evaluated in
Python because the test backend (SQLite) cannot do
`JSONField __contains=[...]`. Wasteful on Postgres where
containment is a first-class operator.

**After.** New helper
`_json_contains_supports_array()` returns
`True` iff `connection.vendor == "postgresql"`. On Postgres
the filter is pushed into SQL; on SQLite / MySQL the rows
are loaded once and filtered in Python. Behaviour is
identical on either backend.

### CI failure fix + real bug in orphan-file walker (`a3a49b51`)

The CI run on `main` after the #1 commit reported
`Required test coverage of 96% not reached. Total coverage: 95.69%`
plus three test failures:

```
FAILED alerts/tests/test_branches.py::test_on_admin_broadcast_swallows_dispatch_error - assert 1 == 0
FAILED alerts/tests/test_triggers.py::test_on_admin_broadcast_dispatches_to_each_recipient - assert set() == {1, 2}
FAILED alerts/tests/test_triggers.py::test_on_admin_broadcast_fans_out_via_celery_group - assert 0 == 1
```

Root cause: my #1 commit changed the call site from
`dispatch_alert` (sync) to `group(...).apply_async()` (async),
but the test suite was still patching the old call target.
The patches were a no-op in CI (where the broker shim does
not raise), but happened to be no-op-safe locally (where the
shim does raise), so the failure only surfaced in CI. The
correct patch target is the `celery.group` symbol used in
`alerts.triggers`, not `dispatch_one_alert.apply_async`.

The same commit also:

- Removed `_save_audio_bytes` (dead code after #6) and its
  test.
- Added the broker-down **success** path test
  (`test_on_admin_broadcast_falls_back_when_broker_down`)
  and the **partial-failure** path test
  (`test_on_admin_broadcast_fallback_partial_failure`),
  which were uncovered.
- Added `test_postgres_path_is_used_when_supported` and
  `test_postgres_path_filters_in_sql` for the
  `_json_contains_supports_array() == True` branch of
  `has_subscription` / `subscribed_users_for_farm`
  (uncovered in the SQLite test backend).
- Added `test_absolute_audio_url_handles_value_error` for
  the defensive `except ValueError` branch in
  `_absolute_audio_url`.

**Real bug in `purge_orphan_audio_files`** (caught by adding
the year/month/day walker test): the original implementation
only descended two levels of the
`audio_alerts/YYYY/MM/DD/` tree. The `FileField` `upload_to`
is three directories deep, so the day-level files were
invisible to the walker and orphan files from the current
or previous day would never be cleaned up. Fixed by
collapsing the three near-identical loops into a single
recursive helper.

**Duplicate `def purge_orphan_audio_files`** — the refactor
accidentally left both the new and the old definitions in
`alerts/tasks.py`; the undecorated plain function was
shadowing the `@shared_task`-decorated one. Fixed.

**Test isolation** — `PurgeOrphanAudioFilesTests` now wipes
`MEDIA_ROOT/audio_alerts/` in `setUp` so the file-system
state does not leak between tests.
