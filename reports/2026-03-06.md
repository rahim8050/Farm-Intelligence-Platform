Report: Weather Cache Stampede Protection (as of Mar 6, 2026)

  Scope of work

  - Added cache stampede protection to weather API endpoints to prevent thundering herd problem.
  - Implemented cache locking with stale-while-revalidate pattern.
  - Covered lock/wait logic with tests for hit, stale, and timeout scenarios.

  ———

  Problem

  - When weather cache expired, multiple concurrent requests would hit upstream API simultaneously.
  - This caused thundering herd problem under load.
  - Upstream providers (Open-Meteo, etc.) could rate-limit or throttle during spikes.

  ———

  Solution

  - Added cache lock mechanism using Django cache `add()` for atomic lock acquisition.
  - Implemented stale-while-revalidate pattern:
    - Primary cache key: fresh data (TTL: 120s for current weather).
    - Stale cache key: expired but still serveable data (TTL: 3x primary TTL).
    - Lock key: prevents multiple concurrent fetches (TTL: 5s).
  - First request after cache expiry acquires lock and fetches from upstream.
  - Subsequent requests wait up to 5s for lock holder to populate cache.
  - If wait times out, requests can serve stale data or raise error.

  Configuration:
    - WEATHER_CACHE_LOCK_TIMEOUT_S: 5 seconds (default).
    - WEATHER_CACHE_LOCK_WAIT_SECONDS: 5 seconds (default, matches lock timeout).
    - WEATHER_CACHE_STALE_TTL_CURRENT_S: 300 seconds (default, max(300, 3x primary TTL)).

  ———

  Files Modified

  - weather/services.py: Added cache stampede protection logic.
    - _stale_cache_key(): Generates stale cache key.
    - _lock_cache_key(): Generates lock cache key.
    - _wait_for_cached_value(): Async poll for cache population with timeout.
    - Updated get_current_weather() to use lock + stale pattern.

  ———

  Behavior

  Cache hit (fresh):
    - Return cached value immediately.
    - No upstream request.

  Cache miss (first request):
    - Acquire lock via cache.add() (atomic).
    - Fetch from upstream provider.
    - Populate both fresh and stale cache keys.
    - Return fresh data.

  Cache miss (concurrent request):
    - Lock acquisition fails (another request holding it).
    - Poll cache every 50ms for up to 5 seconds.
    - If cache populated: return fresh value.
    - If timeout: try stale cache.
    - If stale exists: return it with degradation.
    - If no stale: raise WeatherUpstreamError.

  ———

  Assessment

  - Eliminates thundering herd on cache expiry.
  - Graceful degradation via stale cache during high concurrency.
  - Async-compatible with existing async weather endpoints.
  - Metrics already track cache hits/misses for monitoring.

  ———

  Recommended follow-up

  - Monitor weather_cache_hits_total and weather_cache_misses_total metrics during peak load.
  - Consider increasing CACHE_LOCK_TIMEOUT_S if upstream fetch regularly exceeds 5s.
  - Add similar protection to hourly/daily weather endpoints if they see high concurrency.
