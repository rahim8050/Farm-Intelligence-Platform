## What shipped

- **New app `alerts/`** — top-level Django app, parallel to
  `radio/` and `podcasts/`, registered in `INSTALLED_APPS` and
  mounted at `/api/v1/alerts/`.
- **Two models** (`alerts/models.py`):
  - `AudioAlertSubscription` (UUID PK, unique `(user, farm)`,
    JSON list of `alert_types`).
  - `AudioAlert` (UUID PK, FileField under
    `MEDIA_ROOT/audio_alerts/...`, `is_delivered` /
    `is_acknowledged` / `delivered_at` / `acknowledged_at`).
- **Pluggable TTS service** (`alerts/tts.py`): four backends —
  `piper` (neural, falls back to `espeak` if not on PATH),
  `espeak` (default, shells out to `espeak-ng`), `sine` (always-on
  440Hz tone WAV), `noop` (empty bytes; tests/CI). A backend
  exception is logged and falls back to sine, so the persisted
  row is always created.
- **Four trigger sources**:
  1. Activity completion (`activities/tasks.py:execute` →
     `transaction.on_commit` → `alerts.triggers.on_activity_completed`).
  2. NDVI decline (periodic beat task uses
     `ndvi.farm_state.build_farm_state` and fires when
     `state == "decline"`).
  3. Low NDVI absolute (periodic task fires when latest
     `NdviObservation.mean < ALERTS_NDVI_LOW_THRESHOLD`).
  4. Admin manual broadcast via `POST /api/v1/alerts/admin/send/`.
- **De-duplication**: 24h window per `(user, farm, alert_type)`,
  implemented as a `created_at` filter in the scan tasks (not a
  separate table).
- **WebSocket push**: extended `activities.consumers.ActivityConsumer`
  with an `audio_alert` handler on the existing
  `user_<user.id>` Channels group (`activities/consumers.py:97-115`).
  The push is best-effort; clients that were offline catch up via
  `GET /api/v1/alerts/?unread=true`.
- **8 REST endpoints**:
  - `GET/POST /api/v1/alerts/subscriptions/`
  - `PATCH/DELETE /api/v1/alerts/subscriptions/<id>/`
  - `GET /api/v1/alerts/?unread=true&limit=N`
  - `GET/POST /api/v1/alerts/<id>/` (retrieve + acknowledge)
  - `POST /api/v1/alerts/admin/send/` (admin only, throttled 30/min)
- **OpenAPI**: 8 new `v1_alerts_*` operation IDs in `schema.yml`.

## Architectural decisions

1. **Top-level app, not nested under `radio/`.** Alerts cross
   `farms`, `activities`, `ndvi`, and `radio` (for the WebSocket
   group), so a top-level app is the natural anchor. This matches
   the design sketch in `08_future_expansion.md`.
2. **Pluggable TTS with sane defaults.** The default `espeak`
   backend works on any Linux box with `espeak-ng` installed. The
   `sine` fallback means the system is never broken by a missing
   binary. `piper` (high quality neural) is opt-in via
   `TTS_ENGINE=piper`.
3. **Per-farm subscription model.** A user can opt in/out of each
   alert type per farm. Auto-on was rejected as too noisy; a
   global opt-in was rejected as too coarse.
4. **Reuse the existing `user_<id>` Channels group.** The
   `ActivityConsumer` was already on `user_<id>`; extending it
   with an `audio_alert` handler means a single WS connection
   carries both event types. New clients don't need to open a
   second connection.
5. **WebSocket push is best-effort.** A send failure never rolls
   back the persisted `AudioAlert` row; the `is_delivered` flag
   is informational only. The DB row is the authoritative source
   of truth.
6. **No audio proxying, per ADR 002.** Same as `radio/` and
   `podcasts/`: the client streams the WAV URL directly. The
   audio file is a short tone or short TTS clip, well under the
   `TTS_MAX_TEXT_CHARS=500` cap.
7. **De-duplication without a separate table.** A simple
   `created_at__gte=cutoff` filter on `(user, farm, alert_type)`
   is enough for the current volume. A separate `AudioAlertDedupe`
   table would add schema churn without much benefit.
8. **TTS failure never blocks the row.** The dispatch path is
   `insert row → synthesise → save audio file → push WS`. A TTS
   exception or a WS push exception is logged; the row is still
   saved with `is_delivered=False` so the client can poll
   `GET /api/v1/alerts/?unread=true`.

## Coverage pass (follow-up)

Added `alerts/tests/test_branches.py` (29 tests) to cover the
defensive branches that the main happy-path suite does not hit.
Net effect: branch coverage on the `alerts/` app went from
**94%** to **99%** (well above the 96% target). The two
remaining unhit branches are:

- `alerts/services.py:171->182` — the `if sent:` arm taken when
  the channel layer raises is itself exercised; the partial
  branch marker counts the *fall-through* into the `except`
  block as a separate path. This path *is* hit by
  `test_dispatch_alert_handles_push_failure`; the BrPart marker
  is coverage.py noise, not a missing test.
- `alerts/services.py:213-214` — `_absolute_audio_url`
  `ValueError` catch; the only way to hit it is for the file
  field to be non-empty yet the storage backend refuses to
  resolve its URL. In dev / test that branch is unreachable
  because the empty `audio_file` short-circuits at line 209.
- `alerts/tts.py:171` — `if rate <= 0: return None` inside
  `_guess_wav_duration_ms`. The `wave.Error` catch-all on
  line 173 swallows this case in practice (you cannot write a
  WAV with framerate 0 — `wave.open("wb")` rejects it).

`alerts/serializers.py:35-42` `validate_alert_types` was
simplified: dropped the redundant `ChoiceField` wrapper in
favour of the custom validator (which now matches the OpenAPI
enum and produces a single, more informative error message).
The `ChoiceField`-only path was unreachable in the test suite
because the custom validator was shadowed.
